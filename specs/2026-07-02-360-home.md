# 修复 Home 页面布局，移除多余横向滚动条

## 问题

Home 入口页在内容已经适应可用宽度时，仍出现不必要的横向滚动条。这属于基础 UI 完整性缺陷，会直接削弱首次进入产品时的专业感与稳定感。

从现有 Home/Entry 布局看，问题大概率不是某个明确需要横向滚动的局部组件，而是页面级容器在桌面 viewport 下产生了虚假溢出，最终让 `document.documentElement.scrollWidth` 大于 `clientWidth`。这类回归通常来自以下几类布局约束：

- 顶层 shell 使用 `100vw`，在存在纵向滚动条时把滚动条宽度也算进布局宽度。
- Grid/Flex 子项缺少 `min-width: 0`，被内容最小宽度撑破。
- Home hero、tab rail、recent projects、plugin section 之一在紧凑宽度下没有正确收缩。
- 某一层为了掩盖问题增加了 `overflow-x: hidden`，但根因仍留在布局树里。

## 目标

- Home 页面在内容已适配 viewport 宽度时，不显示页面级横向滚动条。
- 保持当前 Home 信息架构和交互，不把该修复扩展成视觉重构。
- 修复根因，而不是单纯在 `body` 或外层容器上强行隐藏横向溢出。
- 为该问题补一条可重复的回归验证，避免后续 Home/Entry 布局改动再次引入虚假滚动条。

## 非目标

- 不调整 Home 的内容结构、文案、导航关系或组件顺序。
- 不顺手改造 unrelated 的 Entry chrome、Project workspace、Design System flow 布局。
- 不在本次修复里处理真实需要横向滚动的局部区域；局部滚动容器可保留，但不能把页面本身撑出 viewport。

## 方案

### 1. 先写红测，固定回归表现

优先复用现有 Playwright Home/Entry 覆盖，而不是新建一套独立 harness。

- 在 `e2e/ui/entry-chrome-flows.test.ts` 或 `e2e/ui/home-hero-rail.test.ts` 补充/收紧断言。
- 断言口径使用页面级指标：
  - `Math.max(0, document.documentElement.scrollWidth - document.documentElement.clientWidth)`
  - 必要时补充 `body.scrollWidth`、`entry-main`、`home-view`、可疑 rail 容器的宽度差，帮助快速定位。
- 至少覆盖一个会稳定暴露问题的桌面宽度，优先选紧凑 desktop 宽度而不是移动端，因为 issue 描述是 Home 正常桌面布局下的虚假滚动条。

预期结果：

- 基线分支红。
- 修复后绿。

### 2. 定位页面级溢出的真正来源

按 Home 入口渲染链从外到内排查：

- `apps/web/src/styles/shell.css`
  - 重点检查 `.workspace-shell` 这类顶层容器是否使用 `width: 100vw` / `max-width: 100vw`。
- `apps/web/src/styles/home/entry-layout.css`
  - 检查 Entry rail 与主内容 grid 在 rail 展开/收起时是否留下额外宽度。
- `apps/web/src/styles/home/home-hero.css`
  - 检查 hero、type tabs、subtype row、scenario rail 是否在紧凑宽度下把父容器撑宽。
- `apps/web/src/styles/home/plugins-home.css` 与 `recent-projects.css`
  - 检查卡片轨道、gap、padding、min-width 是否存在只在某些组合下触发的溢出。

定位原则：

- 优先修正最上游的宽度定义错误。
- 只有在确认溢出由具体子容器引起时，才在该子容器上补 `min-width: 0`、`max-width: 100%`、收缩规则或局部滚动约束。
- 禁止通过全局 `overflow-x: hidden` 掩盖问题并绕过真实布局修复。

### 3. 实施最小范围修复

修复应保持单页面、单问题闭环。预期最可能的实现形态：

- 若根因是 viewport 单位误用：
  - 把页面级壳层从 `100vw` 改为 `width: 100%` / `max-width: 100%`，让容器跟随可用布局宽度而非物理 viewport 宽度。
- 若根因是 Grid/Flex 子项不收缩：
  - 在具体 Home/Entry 子容器上补 `min-width: 0`、约束 `max-width`，或把 `auto` 轨道改为 `minmax(0, 1fr)`。
- 若根因是某个 rail 或卡片组：
  - 保留局部横向滚动行为，但把页面级溢出限制在组件内部，不让其传播到 `documentElement`。

### 4. 保持验证闭环

完成修复后至少验证：

- 页面级 `scrollWidth - clientWidth <= 2`。
- Home hero 正常可见、可输入。
- 现有局部横向轨道仍可滚动，例如 template carousel 不被错误裁剪。

## 风险

- 把 `100vw` 一类页面级宽度直接改掉，可能影响 workspace/entry 以外的壳层布局，需要确认修复只影响 Home 入口链或验证不会回归其它入口页。
- 给父容器补 `overflow-x: hidden` 很容易让局部菜单、tooltip、popover 被裁掉，所以不能用掩盖式修复替代真实宽度修复。
- Home 有多个可展开的 rail、dropdown、preset card 区域；如果修复点选错，可能让虚假滚动条消失，但破坏真实可滚动区域或内容可见性。

## 验证

### 自动化

- 运行针对 Home/Entry 的 Playwright 用例，至少包括新增或更新的横向溢出回归测试。
- 如修复落在 `apps/web` 样式层，运行：
  - `pnpm --filter @open-design/web typecheck`
- 若变更影响 e2e 断言或使用到 e2e 辅助代码，再补：
  - `pnpm --filter e2e typecheck`

### 人工

- 用 `pnpm tools-dev run web --daemon-port <port> --web-port <port>` 启动 Home 页面。
- 在 issue 复现宽度和一个更宽的桌面宽度下检查：
  - 页面底部不再出现虚假横向滚动条。
  - Home hero、顶部 chrome、recent projects、plugin section 没有被裁切。
  - 局部 carousel/rail 仍可按预期横向滚动。

## 交付范围

- `apps/web/src/styles/home/*` 和/或 `apps/web/src/styles/shell.css` 中的最小必要 CSS 修复。
- 一条针对 Home 页面级横向溢出的回归测试更新。
- 不引入新的路由、状态、接口或跨系统改动。
